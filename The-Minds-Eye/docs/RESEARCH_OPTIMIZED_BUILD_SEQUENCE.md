# Research-Optimized Build Sequence

## Purpose

This build sequence reflects the deep research findings plus a key product correction: the Sanborn Map Builder should be the first major content system after the minimal provenance/source rules are in place.

The Mind's Eye should be built as an approval-driven historical learning operating system with a simulation shell, not as a game that later attempts to recover instructional rigor.

## Guiding Principle

Build the trust layer first, then build the map early.

The map is the opportunity engine. It reveals the buildings, streets, railroads, depots, churches, courts, businesses, creeks, bridges, and civic locations that make accurate learning scenarios possible.

The correct sequence is:

1. minimal provenance and source rules
2. Sanborn Map Builder and city database
3. community review and map approval
4. historical digital twin
5. standards, HQIM, assessment artifacts, and mastery evidence
6. world simulation and domain systems
7. teacher/student workflow
8. visual polish
9. scale and governance

## Phase 0: Product Constitution

Before coding major features, define the operating rules.

Deliverables:

- product principles
- provenance rules
- HQIM alignment framework
- student privacy posture
- teacher authority model
- community review model
- creative licensing labels

Exit criteria:

- every generated object has a provenance strategy
- every lesson has an instructional quality strategy
- every student-facing AI feature has a teacher-control strategy

## Phase 1: Minimal Provenance and Source Foundation

Build first:

- source schema
- citation schema
- provenance schema
- review status model
- audit log
- stable object ID pattern
- verified / inferred / unknown / simulation / creative-license labels

Why:

No map, quest, NPC, artifact, or visual can be trusted without source and provenance infrastructure. But this phase should stay intentionally lean so the project can move quickly into map construction.

Exit criteria:

- system can label verified source, reviewed inference, unknown, simulation element, creative license, and student-generated content
- source integrity table can be generated
- audit records exist for review decisions
- map objects can carry citations and confidence scores

## Phase 2: Sanborn Map Builder and City Database

Build immediately after the minimal provenance layer.

Build:

- Portal to Texas History connector
- Library of Congress Sanborn source support where needed
- Sanborn sheet ingestion
- index sheet detection
- sheet relationship/stitching proposals
- street extraction candidates
- building footprint extraction candidates
- building label extraction candidates
- stable building IDs
- street graph
- draft city database
- unknown-building handling
- source packet builder
- OCR/text metadata handler
- city directory/photo/source enrichment hooks

Why:

The map is the possibility layer. It tells the system what the town actually offers before the system tries to invent lessons, quests, commerce, institutions, railroad tasks, court scenarios, or science connections.

A strong map-first approach changes the lesson generator from:

> What quest can AI invent?

into:

> Given this verified town, what authentic learning opportunities already exist?

Exit criteria:

- system can create a source bundle for one town/year
- sheets can be identified and associated
- draft streets and buildings exist with stable IDs
- unknown buildings are marked honestly
- map data carries provenance and confidence
- draft city database can be sent to review

## Phase 3: Community Review and Map Approval

Build:

- map review interface
- building/street approval workflow
- correction tools
- approval states
- reviewer notes
- sensitive content flags
- consistency checks

Why:

AI should propose the map, but humans must approve the map before it becomes playable.

Exit criteria:

- reviewers can approve/correct/reject map features
- buildings have stable IDs
- streets are clearly named where source-supported
- unknown buildings remain marked unknown
- map package can be finalized
- finalized map package is versioned

## Phase 4: Historical Digital Twin

Build:

- town graph
- streets
- buildings
- institutions
- rail nodes
- commerce nodes
- civic nodes
- bridges/creeks/geographic features
- approved town package

Why:

The digital twin is the approved town model. It becomes the source of truth for simulation, lessons, maps, visuals, and TEKS scenario matching.

Exit criteria:

- teacher can browse approved town data
- system can query town locations by street, building type, institution, railroad feature, civic function, or source
- scenario generator can inspect the approved town inventory

## Phase 5: Standards, HQIM, and Mastery Layer

Build:

- TEKS registry
- subject/course picker
- standard-to-skill mapping
- learning claim model
- evidence requirement model
- HQIM evidence binder structure

Why:

Once the town's opportunity space exists, the system can intelligently match real locations and historical features to TEKS-aligned learning tasks.

Exit criteria:

- teacher can select TEKS
- system can define what counts as evidence for that TEKS
- system can suggest town-based opportunities for a selected TEKS
- lesson package can export standards alignment

## Phase 6: Assessment Artifact and Rubric Layer

Build:

- artifact framework
- rubric schema
- rubric upload/manual creation
- AI feedback draft workflow
- teacher final scoring/override
- mastery attempt ledger

Why:

Students must prove learning through authentic academic products before simulation richness expands.

Exit criteria:

- student can submit artifact
- artifact can connect to TEKS and rubric
- teacher can score/override
- mastery dashboard can update from approved evidence

## Phase 7: World Engine and Domain Systems

Build:

- world state
- event ledger
- travel
- railroad events
- commerce/inventory
- institutions
- justice/court consequences
- NPC state

Why:

Only after the digital twin is approved should the world become interactive.

Exit criteria:

- student/team actions create ledger events
- teacher can approve major consequence branches
- world state changes are traceable

## Phase 8: Experience Engine and Lesson Generator

Build:

- real-world lab connectors
- writing connectors
- scenario generator
- TEKS-to-scenario matcher
- artifact requirements embedded into quests
- teacher approval workflow

Why:

The lesson generator should use approved town data, the assessment infrastructure, and the TEKS/HQIM layer together.

Exit criteria:

- teacher can generate a standards-aligned scenario from approved town data
- scenario requires TEKS execution
- artifact evidence is embedded into gameplay

## Phase 9: Teacher and Student Dashboards

Build:

- teacher command center
- lesson approval screen
- live session controls
- mastery dashboard
- student role view
- map view
- artifact submission
- vocabulary/accessibility supports

Why:

Dashboards should expose trustworthy systems, not hide unstable infrastructure.

Exit criteria:

- teacher can run a class session
- students can navigate, act, submit evidence, and receive supports
- teacher can see mastery status

## Phase 10: Visual Map Renderer and AI Polish

Build:

- deterministic zoomable map renderer
- visual style layer
- optional AI image polish
- provenance/creative badges
- content credentials where possible

Why:

Visual beauty should come after approved data.

Exit criteria:

- map renders approved town package
- labels match approved database
- uncertain locations display uncertainty
- generated visuals are labeled as visualizations

## Phase 11: Multi-Town Scale and Governance

Build:

- town onboarding templates
- reviewer roles
- contributor workflow
- moderation queues
- district export/import
- analytics/observability
- security/accessibility checks

Why:

The product should not depend on one designer or one town.

Exit criteria:

- new towns can onboard using repeatable workflow
- community reviewers can improve town data
- districts can evaluate adoption evidence

## Non-Negotiable

Do not allow the simulation layer to create truth.

Truth comes from sources, review, teacher approval, and explicit creative labels.

The map comes early because it reveals what accurate instructional possibilities exist, but the map still must carry provenance and pass human review before becoming the playable world.
