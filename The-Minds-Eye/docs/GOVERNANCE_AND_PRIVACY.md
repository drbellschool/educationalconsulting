# Governance and Privacy

The Mind's Eye must preserve historical truth while protecting students and respecting local community concerns.

## Historical Moderation Principle

Do not silently erase historical records.

The product should allow communities and districts to:

- review
- contextualize
- flag
- age-band
- temporarily quarantine
- recommend teacher-only presentation

But original source provenance must remain intact.

## Content Layers

### 1. Source Layer

The immutable archival layer.

Includes:

- source ID
- title
- date
- URL
- OCR text when available
- image/map reference
- archive/provider
- citation metadata

### 2. Instructional Layer

Teacher-facing and student-facing presentation.

Includes:

- simplified excerpts
- vocabulary supports
- content warnings
- lesson framing
- sensitive topic notes
- student-safe presentation

### 3. Community/District Policy Layer

Local presentation rules.

Includes:

- grade-band visibility
- reviewer notes
- caution labels
- district restrictions
- parent/community concern notes
- approval state

## Required Audit Log

Every moderation action should record:

- action ID
- source/content ID
- reviewer ID
- reviewer role
- timestamp
- action taken
- reason code
- affected grade bands
- whether teacher view changed
- whether student view changed
- appeal status

## Sensitive Content Categories

Potential categories:

- racial violence
- discriminatory language
- graphic violence
- sexual content
- child safety concern
- political extremism
- religious conflict
- local family sensitivity
- criminal accusation
- unverified claim
- living person concern

## Student Data Privacy

Design around data minimization.

Collect only what is needed for instruction and reporting.

### Student Data Types

- student ID or roster ID
- class membership
- TEKS evidence attempts
- mastery status
- teacher comments
- accessibility settings
- generated role/avatar choice

### Avoid Unless District Approved

- real student photos
- biometric-like identity data
- unnecessary chat retention
- public display of student performance
- community access to student data

## Student Avatars and Images

The platform should work without real student faces.

Default avatar options:

- illustrated historical-style persona
- student-selected traits
- random generated role portrait
- teacher-approved class-safe avatar pack

If photo-based avatars are ever enabled:

- opt-in only
- district-controlled
- parent/guardian consent where required
- short retention window
- original image deletion after generation when possible
- no community dashboard access
- teacher/admin controls

## FERPA/COPPA Posture

The product should be designed for school-controlled educational use.

Implementation should support:

- district data agreements
- school official model where applicable
- role-based access control
- student data deletion/export
- retention rules
- parent/school notice support
- no advertising use of student data

## AI Safety

Use moderation for:

- student chat
- teacher prompts
- generated mission text
- generated images
- avatar prompts

The system should log blocked output without exposing harmful material to students.

## Non-Negotiables

1. Community reviewers cannot rewrite primary sources.
2. Teachers approve lessons before student launch.
3. Students should see source-grounded explanations, not hallucinated history.
4. AI-generated fiction must be labeled.
5. Student privacy must be designed in from the first prototype.
