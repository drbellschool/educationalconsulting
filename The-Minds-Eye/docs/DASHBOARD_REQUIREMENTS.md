# Dashboard Requirements

The platform needs three primary dashboards:

1. Community Dashboard
2. Teacher Dashboard
3. Student Dashboard

## 1. Community Dashboard

### Purpose

The community dashboard lets approved local historians, museums, archives, school partners, and district leaders review how the town's history is presented.

It should protect students without rewriting or erasing the historical record.

### Core Features

- town profile management
- historical people directory
- historical place/building directory
- source review queue
- community context notes
- sensitive-content flags
- age-band visibility recommendations
- local advisory notes
- appeals workflow
- audit log

### Moderation Philosophy

Use review, contextualization, age-gating, and teacher warnings instead of silent censorship.

The original source remains preserved. The community layer can add:

- warning labels
- explanation notes
- alternative excerpts
- grade-band limits
- teacher-only visibility
- review status

### Suggested Review States

- approved
- approved with context
- teacher-review required
- age-restricted
- temporarily quarantined
- rejected for classroom use
- needs source verification

## 2. Teacher Dashboard

### Purpose

The teacher dashboard is the instructional command center.

The teacher should be able to request a lesson, approve sources, launch the simulation, manage students, and track mastery.

### Lesson Builder Inputs

- grade/course
- subject
- TEKS selection
- town
- year or date range
- class size
- session length
- reading level
- EB support level
- 504 accommodations
- SPED accommodations
- number of mastery demonstrations required
- whole-class, teams, or individual mode

### Teacher Approval Workflow

Before launch, the teacher should see:

- selected historical moment
- source packet
- map locations
- verified facts
- inferred context
- fictional gameplay additions
- sensitive content warnings
- standards alignment
- generated student materials

The teacher can approve, regenerate, edit, or reject.

### Live Class Controls

- start/pause/end session
- move all groups to next checkpoint
- spotlight a team response
- trigger whole-class dice roll
- trigger team dice roll
- lock/unlock student chat
- send teacher announcement
- open quick poll
- mark evidence as mastery
- override AI scoring

### Mastery Dashboard

For each TEKS/student, show:

- attempts
- successful demonstrations
- required demonstrations
- evidence artifacts
- teacher notes
- current status

Suggested statuses:

- not started
- attempted
- developing
- demonstrated once
- mastery met
- needs reteach

## 3. Student Dashboard

### Purpose

The student dashboard should be simple and immersive.

Students should not manage the whole system. They should focus on their role, evidence, vocabulary, decisions, and reflection.

### Core Student Features

- role card
- character identity or avatar
- mission briefing
- map/location view
- evidence notebook
- vocabulary coach
- translation support
- sentence frames
- dice panel
- team chat or response space
- mastery prompts
- reflection exit ticket

### Student Role Types

- investigator
- reporter
- cartographer
- engineer
- witness interviewer
- historian
- council delegate
- business owner
- town planner
- scientist
- data analyst

### Large Group Model

For 20-30 students, do not run 30 totally separate adventures.

Use:

- teacher-led launch
- teams of 3-5
- rotating student roles
- shared class timeline
- checkpoint-based progression
- team evidence submissions
- individual mastery checks

The AI should orchestrate groups while the teacher controls pace and authority.
